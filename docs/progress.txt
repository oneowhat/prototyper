## 2026-07-10 — Task: Scaffold Python package structure
Changed: Added pyproject.toml (hatchling, `prototyper` console script,
  runtime deps jinja2/weasyprint/pyyaml declared for later tasks),
  src/prototyper/{__init__.py,__main__.py,cli.py}, and
  tests/test_scaffold.py covering version, CLI entry, and `python -m`.
Why: Named the package `prototyper` to match the `.prototyper/` history
  folder convention used throughout the PRD. Chose a src/ layout so tests
  run against the installed import path, not the repo root. The scaffold
  is deliberately stdlib-only (argparse, no click) so the package imports
  and the CLI runs before rendering deps are installed; the build/watch/
  note subcommands are registered (stable CLI surface) but raise
  NotImplementedError until their tasks land. pytest is configured via
  pyproject (pythonpath=src) so `pytest` works with no install step.

## 2026-07-10 — Task: Define project.yaml schema and loader
Changed: Added src/prototyper/config.py (frozen dataclasses
  ProjectConfig/ComponentSize/Layout, ConfigError, load_project loader)
  and tests/test_config.py covering valid preset/custom configs, layout
  defaults + overrides, path resolution, and validation errors.
Why: Modeled the schema straight from the PRD's project folder convention
  (component size, data path, template path, layout hints) as immutable
  dataclasses so downstream stages get a stable, typed contract. Kept the
  loader strictly structural: it resolves data/template paths against the
  project dir but does NOT check file existence (job of the CSV/template
  loaders) and does NOT resolve size presets or parse dimension units
  (deferred to the component-sizing task) — component size is captured
  verbatim (preset xor width+height, enforced) so those steps have
  well-formed input. load_project accepts either the project directory or
  the project.yaml path for CLI ergonomics, unknown keys are rejected to
  surface typos early, and every failure path raises ConfigError with a
  human-readable message rather than a raw YAML/KeyError. Layout defaults
  (0.5in margin, 0.1in gutter, cut_lines on) chosen for home-printing
  safety and PRD alignment.

## 2026-07-10 — Task: Implement CSV data loader
Changed: Added src/prototyper/data.py (frozen Dataset dataclass with
  fieldnames/rows + __len__, DataError, load_data loader) and
  tests/test_data.py covering row parsing, header/value normalization,
  BOM/UTF-8/blank-line/short-row tolerance, and every error path.
Why: Modeled the output as the PRD's input contract — an ordered list of
  rows, one per component instance, each a {header: value} mapping — so
  the renderer can instantiate one component per row and resolve
  placeholders by column name. Values stay strings because CSV is untyped
  and formatting is the template's job. Used csv.reader (not DictReader)
  to keep full control over header validation and jagged-row handling.
  The loader is strict about corruption that silently ruins a print run
  (duplicate/blank headers = ambiguous placeholders, rows wider than the
  header = misaligned data, header-only files = nothing to build all
  raise DataError) but forgiving about harmless spreadsheet artifacts
  (utf-8-sig strips Excel's BOM, trailing empty cells pad to "", blank
  lines skip). Errors carry the 1-based line number for hand-editing.

## 2026-07-10 — Task: Implement single-component HTML/CSS rendering
Changed: Added src/prototyper/render.py (RenderError + render_component,
  which turns one Jinja2 HTML/CSS template plus one {header: value} data
  row into the substituted HTML for a single component) and
  tests/test_render.py covering substitution, autoescaping, filters/
  conditionals, relative includes, and every error path. Installed jinja2
  (already a declared dependency) into the dev environment.
Why: This is the smallest rendering unit the PRD's pipeline needs — "one
  component defined once, instantiated once per row" — deliberately scoped
  to produce HTML for a single component; page packing and PDF assembly are
  later tasks. Row keys are passed as top-level template variables so the
  PRD's documented {{ name }} placeholder style works directly. Two strict
  defaults keep it consistent with the config/data loaders' "fail loud on
  anything that silently ruins a print run" stance: StrictUndefined makes a
  typo'd placeholder ({{ nmae }}) raise instead of printing a blank card
  (optional fields remain expressible via the standard `| default('')`
  filter, which still works under StrictUndefined), and autoescaping is on
  so data values containing <, & or " can't emit broken HTML — the template
  supplies markup, data supplies text. The Jinja2 Environment uses a
  FileSystemLoader rooted at the template's directory so {% include %}/
  {% extends %} resolve relative to it. Template syntax errors and unknown-
  field references are both wrapped in RenderError with a human-readable
  message (syntax errors carry the line number).

## 2026-07-10 — Task: Implement component size presets and custom sizing
Changed: Added src/prototyper/sizing.py (ResolvedSize dataclass, SizeError,
  the _PRESETS table, parse_length/resolve_preset/resolve_size/list_presets)
  and tests/test_sizing.py covering preset resolution + name normalization,
  unit parsing/conversion, and every error path.
Why: This is the step the config loader deliberately deferred — it captured
  component size verbatim (a preset name xor raw width/height strings) so
  this module could turn it into concrete numbers the packing/rendering
  stages compute with. Canonicalized to inches (float) because the output
  page is 8.5x11in, making inches the natural basis for grid-packing math;
  parse_length converts the CSS absolute units (in/mm/cm/pt/pc/px) and
  treats a bare number as inches for ergonomics. Shipped the PRD's example
  presets (poker/bridge/tarot + mini, hex-token, square-token); ResolvedSize
  carries a `shape` ("rect"/"hex") so hex tokens are meaningfully distinct
  from square ones and downstream cut-line/packing code has an outline to
  key on — custom sizes are always rectangular since the config exposes no
  shape override. Preset names are normalized (case/space/underscore/hyphen
  insensitive) so "Hex Token" == "hex-token". Kept the loaders' fail-loud
  stance: unknown preset (lists the valid ones), unparseable length, unknown
  unit, and non-positive dimensions all raise SizeError with a clear message.
  Fit-on-page validation is intentionally left to the packing engine.

## 2026-07-10 — Task: Build auto-grid page packing engine
Changed: Added src/prototyper/pack.py (PAGE_WIDTH_IN/PAGE_HEIGHT_IN
  constants, Placement/Sheet/PackedLayout dataclasses, PackError, and
  pack_components) and tests/test_pack.py covering grid computation, the
  exact-fit floating-point boundary, placement coordinates, multi-sheet
  overflow, forced page breaks, and every error path.
Why: This is the PRD's auto-grid page layout — "compute how many components
  fit per 8.5x11 sheet and fill sheets in data order". Kept it pure
  geometry: it takes a component *count* (not the data rows or rendered
  HTML) plus a ResolvedSize and numeric margin/gutter, and returns
  positioned trim boxes. Decoupling packing from data/render means it is
  testable with plain numbers and lets the cut-line and PDF-assembly stages
  each iterate Placements without re-deriving the grid. Placements carry
  their own width/height so each is a self-contained "draw this rectangle
  here" instruction. Grid count is floor((usable+gutter)/(size+gutter))
  with a 1e-6in epsilon so a component that fills the usable area to an
  exact integer number of columns isn't lost to float drift. Chose top-left
  anchoring at the margin (not centring the grid in the usable area) so the
  designer's configured margin is honoured exactly and cut-line coordinates
  stay predictable; leftover space collects at the right/bottom. Packing is
  by bounding box, so hex tokens pack like their enclosing rectangle and
  shape is left for cut-line rendering to interpret. Margin/gutter are
  accepted as inches (the build command will resolve the config's length
  strings via sizing.parse_length). This module also discharges the
  fit-on-page validation the sizing task deferred: a component too wide,
  too tall, or crowded out by an oversized margin raises PackError with a
  message naming the offending dimension and the usable space it must fit.
  Added optional page_breaks (data indices that force a fresh sheet) to
  cover the PRD's forced-page-break layout hint at the engine level, ahead
  of any config surface for it; a break at index 0 or a page boundary is a
  harmless no-op. count=0 yields zero sheets while still reporting the grid.
